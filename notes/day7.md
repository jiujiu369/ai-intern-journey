#	核心理论
1. 	向量数据库是什么？
-	专门用于存储、索引、检索高维语义向量的专用数据库，是RAG系统的核心存储组件
	**与传统的数据库的差别**
- 	MySQL：存文本/数字，只做 精准、关键词字面匹配
- 	向量数据库：存 Embedding 高维向量，做 语义相似度匹配
2. 	为什么关键词检索做不了 RAG？
	关键词检索：字对字匹配，不懂同义句、口语化、改写句
	示例：
- 	用户问：报销几天内交？
- 	知识库原文：出差结束 15 个工作日内提交
- 	关键词检索：匹配不到
- 	义检索：向量空间相似，精准命中
3. 	ChromaDB 离线优势
- 	纯本地文件存储，无需部署服务
- 	自动持久化，重启不丢数据
- 	零外网依赖，完美适配公司内网
4. 	完整 RAG 前置链路
	**Day5 文本清洗 → Day6 本地向量化 → Day7 向量入库 + 语义检索 → 后续大模型生成答案（完整 RAG）**

	**文本分块**
-	核心目的：适配模型长度、提升检索精度、防止语义断裂、控制算力与存储
	1.	适配 Embedding 模型输入长度限制：合理分块，避免输入太长文本导致丢失后半段文本
	2.	缩小检索粒度，提升匹配精准度：一份文档同时包含多类信息（报销、考勤、年假），整篇作为一条向量会混杂大量无关内容。小块只承载单一知识点，用户提问只会召回高度相关片段，不用人工在长文本里翻找
	3.	**滑动重叠解决语句 / 条款被拦腰切断**：若无重叠分割，完整句子、法律条款、业务规则会被拆到两块，单独检索任意一块都会缺失关键信息；设置重叠后，完整语义会落在至少一个分块内
	4.	**平衡向量库存储、检索速度、算力消耗:**
		-	块过小：碎片极多，向量数量爆炸，查询变慢、冗余结果多；
		-	块过大：单块语义混杂，匹配精度暴跌；
		-	分块是为了在「向量数量」和「单块语义纯度」之间找到平衡点
	5.	适配大模型上下文窗口:检索出来的片段要塞进 LLM 提示词，LLM 也有上下文长度限制，分块后的短片段方便筛选少量内容送入模型，避免超长 prompt 超限、推理成本飙升

#	检索 top_k 取值行业标准
-	内部制度 / 人事文档：top_k=2~3，少量片段即可覆盖完整答案，减少大模型干扰	
-	技术手册 / 专业文档：top_k=3~5，知识点分散，多条片段拼接完整信息
-	对外客服通用知识库：top_k=4~6，搭配重排模型过滤低相似度内容
-	不建议 top_k≥8：大量低相关文本污染 Prompt，推理速度下降、易产生幻觉
-	配套优化：增加相似度阈值过滤（一般 0.6~0.7），低于阈值直接丢弃

	2.	接收用户输入文件路径，自动定位文件
	3.	调用 Day5 工具解析文档、清洗文本、滑动分块
	4.	生成元数据，批量向量化入库
	5.	循环交互式语义检索，输入 exit 退出程序
