##  RAG 完整链路 ##
	原始文档 → 文档解析（读内容） → 文本清洗（去噪声） → 文本分块 → 向量化 → 向量库存储 → 检索匹配 → LLM 生成答案
	文档解析 + 文本清洗 的核心作用

-	文档解析作用
		把 PDF/Word/TXT 这种结构化文件，转换成 纯文本字符串。
		大模型、向量数据库只能处理文本，无法直接读取文件格式。
		没有解析 = RAG 完全无法读取知识库。
-	文本清洗作用
		原始解析出来的文本充满无效噪声，直接向量化会生成大量废向量。
		清洗的核心：保留语义、删除噪声，大幅提升检索准确率
##	四种文档格式：解析原理、优缺点、适用场景、踩坑点
1. ## TXT / MD
-	解析原理:
		Python 直接读取文本流，无需解析格式，纯文本输出。
-	优点:
	1.	速度最快、零格式干扰
	2.	清洗成本最低、最干净
-	缺点:
		无法承载图文、表格、复杂排版
-	适用场景
		日志文件、设备记录、开发文档、Readme、极简知识库（物联网设备日志首选）
-	踩坑点
	1.	Windows 文件默认 GBK 编码，直接读必乱码
	2.	部分日志文件含隐藏控制字符
2. ## PDF（企业最常用）
-	解析原理
		逐页读取页面文本图层，拼接成全文。
-	优点	
	1.	企业手册、技术文档、规范文档几乎都是 PDF
	2.	排版固定、内容权威
-	缺点
	1.	扫描版 PDF 是图片，完全无法提取
	2.	原生 PDF 自带大量换行、断句、页码乱排版
-	适用场景
		产品手册、技术白皮书、行业规范、设备说明书（你的物联网项目主力格式）
-	踩坑点（高频报错）
	1.	扫描 PDF 解析为空
	2.	每页单独换行，句子被强行截断
	3.	页眉页脚、页码大量噪声
	4.	加密 PDF 读取失败
3. ## DOCX Word 文档
- 	解析原理
	1.	读取 docx 的段落对象，逐段拼接文本。
- 	优点
	1.	企业制度、项目方案、调研报告常用
	2.	段落结构清晰
-	缺点
	1.	不支持老.doc 格式
	2.	表格、图片无法解析
	3.	排版复杂时文本错乱
-	适用场景
		项目文档、需求文档、公司制度
-	踩坑点
	1.	.doc 直接报错
	2.	空白段落多，产生大量空行噪声


##	三、文本噪声类型、清洗规则、对 RAG 的影响机制（核心难点，面试高频）
-	六大真实 RAG 文本噪声
	1.	不可见控制乱码：系统隐藏字符、转义残符
	2.	冗余空白噪声：多空格、多制表符
	3.	连续空行噪声：页面分隔产生大量空行
	4.	页眉页脚 / 页码噪声：每页重复出现无效文字
	5.	无效特殊符号：水印、装饰符号、标记符号
	6.	断句错乱噪声：PDF 自动换行导致句子被拆分
-	对应清洗规则（今天写的 6 个函数，一一对应）
	1.	清理控制字符
	2.	统一空白格式
	3.	删除连续空行
	4.	过滤页码、版权、页眉
	5.	过滤无效特殊符号
	6.	段落规整、修复断句
-	【最关键】噪声如何毁掉 RAG 检索？
-	机制 1：向量相似度是「字符级、语义级密集匹配」
		脏文本里每一个无效字符、多余空格、乱码，都会生成独立向量。
-	结果：
		有效语义向量被稀释
		噪声向量占据向量库空间
		相似度计算跑偏
-	机制 2：脏文本会导致「相似文本向量不相似」
-	例子：
		干净文本：ESP32设备温度过高触发保护
		脏文本： ESP32 设备 温度 过高 触发保护 \n \n 第12页
		两段语义一模一样，但字符分布完全不同
		向量空间距离变远
		用户提问搜不到正确内容（RAG 最大通病）
-	机制 3：空行、乱码会破坏分块逻辑
		RAG 是按固定长度分块。
		大量空行、垃圾字符会导致：
		有效文本被挤压成碎块
		一块文档全是噪声
		语义被截断、上下文断裂
-	机制 4：重复页眉页脚导致「重复向量爆炸」
		每页都有相同页眉
		系统会认为这是高频核心语义
		检索时优先返回垃圾页眉，彻底干扰结果
